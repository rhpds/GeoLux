THEORETICAL FOUNDATIONS

These foundations are not background context. They are
load-bearing constraints that shape every architectural
decision. Every component specification must reference
the theoretical foundation it responds to.

Computational Irreducibility
The behavioral state space of complex distributed
infrastructure cannot be analytically predicted faster
than the system itself executes. This property makes:
- Closed-form prediction inadequate
- Universal threshold calibration unreliable
- Purely synthetic test generation epistemically incomplete
- Long-horizon planning unsafe without bounded validation

Every architectural component is a deliberate response to
a specific dimension of this irreducibility. Document which
dimension each component addresses and why the component
is necessary given that dimension. The complexity of the
architecture is justified by and proportional to the
irreducibility of the problem domain.

Geometric State Theory
The probability manifold of an LLM has measurable geometric
shape. Stable outputs produce stable geometry. Adversarial
pressure, hallucination, and novel input distributions
destabilize the manifold. This instability is physically
measurable.

The platform uses geometric stability as a first-class
signal across all LLM call sites. Geometric stability is
measured via activation variance across transformer layers
during inference. High variance indicates manifold
instability. Low variance indicates stability.

Geometric stability produces four evaluation states:
- Stable pass: correct output from stable geometry.
  High confidence. Trust the result.
- Unstable pass: correct output from unstable geometry.
  Fragile. Flag for additional validation.
- Stable fail: incorrect output from stable geometry.
  High confidence failure signal. The rubric is correctly
  identifying a real problem.
- Unstable fail: incorrect output from unstable geometry.
  Possible classification artifact. Investigate before acting.

Document how geometric stability is measured, stored,
surfaced, and consumed by each component that calls an LLM.

COMPONENT 1: THE HYPOTHESIS ENGINE

Purpose: Replace LLM-as-decision-maker with
LLM-as-hypothesis-generator. The LLM generates structured
falsifiable hypotheses about observed system state. A
separate deterministic mechanism validates hypotheses
against evidence. The LLM's confidence is irrelevant.
Only evidence validation matters.

Theoretical grounding: Responds to computational
irreducibility by replacing prediction with structured
falsifiable inquiry. Makes agent reasoning falsifiable
in a way that prompt-response systems cannot be.

Inputs:
- Structured evidence bundle from the evidence collection
  layer: cluster state, telemetry, rubric evaluation
  results, event stream data
- Historical hypothesis outcomes: which hypotheses were
  validated, which were falsified, which were inconclusive
- Geometric stability score from the LLM call that
  generated each hypothesis

Outputs:
- Ranked hypothesis queue: ordered by geometric stability
  of originating LLM call, then by prior validation rate
  of similar hypotheses
- Each hypothesis: structured claim about system state,
  testable conditions that would validate or falsify it,
  confidence score, geometric stability score, timestamp
- Hypothesis audit record: persisted to PostgreSQL with
  full evidence snapshot at time of generation

Hypothesis generation process:
- High geometric stability hypotheses generated first
- Low geometric stability hypotheses generated only if
  high stability hypotheses are exhausted or falsified
- Novel failure modes that produce no hypothesis
  candidates trigger a rubric extension event: log the
  unclassified evidence bundle and flag for human review
- Hypothesis queue is consumed by the constraint
  classification layer for validation

Geometric stability integration:
- Measure activation variance across Granite transformer
  layers during each hypothesis generation call
- Store stability score alongside hypothesis in PostgreSQL
- Stable geometry hypotheses weighted higher in queue
- Unstable geometry hypotheses flagged in audit trail
- If geometric stability falls below configurable
  threshold: do not generate hypothesis, log the
  instability event, escalate to human review queue

Interface:
- REST API: POST /hypotheses/generate
  Input: evidence bundle JSON
  Output: ranked hypothesis queue JSON
- WebSocket: hypothesis_generated event
- Kafka topic: hypothesis.generated
- PostgreSQL: hypotheses table with full audit fields

Failure modes:
- LLM unavailable: queue last known hypotheses with
  staleness flag, alert operator
- All hypotheses falsified: trigger rubric extension
  event, escalate to human
- Geometric instability sustained: suspend hypothesis
  generation, alert operator, log duration of instability

COMPONENT 2: EVIDENCE-BASED CONSTRAINT CLASSIFICATION

Purpose: Classify system state by evaluating structured
evidence against formal constraint definitions.
Classification is deterministic given the evidence.
The LLM's role if any is evidence interpretation at
the boundary, not classification. Every classification
decision has a complete evidence chain that can be
traced, audited, and challenged.

Theoretical grounding: Responds to computational
irreducibility by grounding classification in observed
state rather than anticipated state. Responds to geometric
state theory by incorporating stability scores into
classification confidence.

Constraint schema:

- stage: which rubric stage this constraint belongs to
- assertion_type: threshold, boolean, range, pattern,

- assertion_definition: the formal assertion over

- evidence_requirements: which evidence fields must be
  present for this constraint to be evaluable
- severity: critical, major, minor
- remediation_class: which remediation action class
  applies when this constraint fails
- geometric_stability_weight: how much geometric
  stability of the classifying LLM call affects
  confidence in this constraint's evaluation

Current constraint stages (Stargate's eleven rubric stages):
- cluster-health
- namespace-ready
- deployment-ready
- route-ready
- vm-runtime-ready
(document remaining six stages with full constraint

Classification process:
- Evidence bundle received from collectors
- Each constraint evaluated against evidence
  deterministically where possible
- LLM interpretation called only for constraints that
  require semantic understanding of unstructured evidence
- Geometric stability measured for each LLM call
- Classification result: pass, fail, inconclusive,
  unclassifiable
- Confidence score: combines evidence completeness,

Failure modes:

- Constraint schema version mismatch: reject with

- Geometric instability during LLM interpretation:

COMPONENT 3: LLM-MPC (LARGE LANGUAGE MODEL MODEL
PREDICTIVE CONTROL)

Purpose: Apply model predictive control theory to
agent decision-making using LLM reasoning over
system state as the dynamics model. Replaces the
parametric dynamics model of classical MPC with
LLM reasoning over structured evidence. Maintains
the receding horizon, optimization, and replanning
properties of classical MPC.

Theoretical grounding: Responds to computational
irreducibility by using short receding horizons
calibrated to locally smooth dynamics rather than
assuming long-horizon predictability. Responds to
geometric state theory by using geometric stability
as an automatic horizon limiter.

Control structure:
- Maintain a model of current system state from
  evidence collectors
- At each control cycle: generate predictions over
  receding horizon N using LLM reasoning
- Optimize control action against predictions within
  defined constraint boundaries
- Execute first action only
- Observe outcome, update state model, replan
- Repeat

Receding horizon management:
- Default horizon N: configurable, start at 2 steps
- Horizon automatically shortens when geometric
  stability of prediction LLM call falls below
  configurable threshold
- Horizon extends only when sustained geometric
  stability is observed across multiple consecutive
  prediction cycles
- Maximum horizon: configurable, never exceed without
  explicit operator approval
- Minimum horizon: 1 step always

Geometric stability integration:
- Measure stability of each prediction call
- High stability: trust prediction, use for optimization
- Low stability: shorten horizon, flag prediction,
  require higher confidence threshold before action
- Sustained instability: suspend MPC, fall back to
  reactive classification only, alert operator

Objective function:
- Define formally what on-course looks like for each
  cluster and workload type
- Objective is cluster-specific: calibrated from
  operational history via NanoObs and Deepfield
- Objective must be expressible in a form that LLM
  can reason about without ambiguity
- Objective updates are versioned and audited

Optimization step:
- Generate candidate control actions from constraint
  classification layer
- Score each candidate against predicted state trajectory
- Select highest scoring candidate within constraint
  boundaries
- Log optimization decision with full candidate set
  and scores

Interface:
- REST API: POST /mpc/plan
  Input: current state, objective, horizon
  Output: recommended action with prediction trace
- Kafka topic: mpc.action.recommended
- PostgreSQL: mpc_cycles table with full prediction
  and optimization audit trail

Operational history requirement:
- LLM-MPC must not be activated on a cluster until
  that cluster has accumulated minimum configurable
  weeks of rubric evaluation history
- Activation gate enforced programmatically not by
  policy
- Attempting to activate before history threshold
  met returns explicit error with history deficit

Failure modes:
- Prediction horizon collapses to zero: suspend MPC,
  alert operator, log instability duration
- Objective function undefined for cluster: refuse
  to plan, require operator to define objective first
- All candidate actions violate constraints: escalate
  to human, log constraint violation set

COMPONENT 4: DEEPFIELD

Purpose: Govern agent cognitive complexity by mapping
task complexity to hardware substrate as a policy
decision. Task complexity determines reasoning
requirement. Reasoning requirement determines substrate.
Substrate determines cost, latency, and governance
surface. Policy governs the mapping. Different substrates
produce qualitatively different epistemic outputs not
just quantitatively faster ones.

Theoretical grounding: Responds to computational
irreducibility by acknowledging that each cluster traces
a unique irreducible trajectory requiring per-cluster
calibration rather than universal thresholds. Responds
to geometric state theory by using geometric stability
as a routing confidence signal.

Tier definitions:
Nano tier:
- Substrate: CPU
- Agent type: deterministic, rule-based, boolean
  evaluation
- Task characteristics: single assertion, no reasoning
  required, binary outcome
- Governance surface: minimal, outcome is deterministic
- Audit requirement: log input, output, timestamp
- Example: is pod running, is route accessible,
  is replica count at minimum

Micro tier:
- Substrate: Xeon6
- Agent type: moderate reasoning, structured inference
- Task characteristics: multi-condition evaluation,
  pattern matching, structured classification
- Governance surface: medium, outcome is probabilistic
  but bounded
- Audit requirement: log input, output, confidence,
  reasoning summary, geometric stability score
- Example: classify failure mode from pod logs,
  evaluate namespace health against rubric stage

Macro tier:
- Substrate: Gaudi 3
- Agent type: complex reasoning, long context,
  multi-step inference
- Task characteristics: novel failure analysis,
  hypothesis generation, root cause reasoning,
  cross-cluster pattern detection
- Governance surface: maximum, outcome is probabilistic
  and unbounded
- Audit requirement: full evidence bundle, hypothesis
  set, confidence scores, geometric stability profile,
  human review flag if stability below threshold
- Example: executive readiness summary, cross-cluster
  failure correlation, novel failure mode classification

Routing policy:
- Workload classification: determine tier based on
  task complexity characteristics
- Hardware availability check: is target substrate
  available and within utilization threshold
- Fallback policy: if target substrate unavailable
  or saturated, what is the escalation path
- Routing decision logged to audit trail with
  classification reasoning and geometric stability
  score of classification call
- Policy override: operator can manually route with
  mandatory reason code logged

NanoObs integration:
- NanoObs observes nano agent execution against
  current thresholds per cluster
- Evidence accumulates in structured form per cluster
- THE generates hypotheses about threshold drift
- Constraint classification validates whether drift
  is significant enough to warrant adjustment
- Adaptive threshold recommendation produced
- Human approval required before threshold adjustment
  applied to production routing
- All adjustments versioned and audited

Geometric stability in routing:
- Classification call that determines workload tier
  has geometric stability measured
- Stable classification: route with full confidence
- Unstable classification: flag, require secondary
  validation, consider escalating to next tier for
  safety margin
- Sustained instability in classification: suspend
  adaptive routing, fall back to static policy, alert

Agentic coding extension (future phase):
- New nano agents can be proposed by the system in
  response to observed cluster conditions
- Proposal goes through constraint classification
  validation against the cluster's evidence profile
- Synthetic scenario replay against candidate agent
  in sandbox
- If passes: promote to production with audit record
- If fails: reject with full failure evidence record
- No new nano agent touches real cluster state without
  passing the promotion gate
- Document the promotion gate specification fully

Interface:
- REST API: POST /deepfield/route
  Input: workload description, current cluster state
  Output: tier assignment, substrate recommendation,
  routing confidence, geometric stability score
- Kafka topic: deepfield.routing.decision
- PostgreSQL: routing_decisions table with full audit

DEVELOPMENT METHODOLOGY

The platform is built using a layered correctness
discipline. Every component must be built using
all applicable layers. Document the methodology
artifacts required for each component.

EDD -- Evaluation Driven Development:
The primary development methodology. Write the
evaluation rubric before implementation. Build
until the rubric passes. The rubric matrix has:
- Rows: evaluation dimensions covering correctness,
  performance, governance compliance, evidence
  completeness, geometric stability
- Columns: implementation milestones
- Third axis: geometric stability state at each
  evaluation point: stable pass, unstable pass,
  stable fail, unstable fail

Development target: stable green across all
evaluation dimensions. Unstable passes are not
acceptable for production promotion regardless
of passing score.

For each component produce:
- Complete evaluation rubric matrix before any
  implementation guidance
- Geometric stability evaluation criteria
- Promotion gate: what stable green across all
  dimensions looks like for this component

TDD -- Test Driven Development:
Unit and integration correctness layer. Applied
within each EDD milestone. Red/green cycle.

For each component produce:
- Unit test specifications for all pure functions
- Integration test specifications for all
  cross-component interactions
- Test data requirements

BDD -- Behavior Driven Development:
Behavioral correctness layer. Given/When/Then
scenarios that become executable specifications.

For each component produce:
- BDD scenarios for all primary use cases
- BDD scenarios for all failure modes
- BDD scenarios for all governance properties

PDD -- Property Driven Development:
Invariant correctness layer. Define properties

For each component produce:

  always hold regardless of input
- Property test specifications using Hypothesis

SBE -- Specification by Example:
Concrete examples that serve simultaneously as

For each component produce:

CDD -- Contract Driven Development:
Interface contract correctness layer. Define API

independently against contracts.

For each component produce:

- AsyncAPI specification for all Kafka topics
- Pact contract tests for all consumer/provider pairs
- Contract version management approach

GDD -- Geometric Driven Development:
Novel methodology layer specific to this platform.
Geometric stability of LLM calls is a first-class

For each component that calls an LLM produce:

GEOMETRIC STABILITY MEASUREMENT SPECIFICATION

This is a novel component with no existing implementation
to reference. Specify it completely.

Measurement approach:
Primary: activation variance across transformer layers
during LLM inference. Requires access to model
internals. Available when running Granite locally
via vLLM on Gaudi or Xeon6.

Fallback when activation access unavailable:
- Output token probability variance across top-k
  alternatives at each generation step
- Perplexity of generated output against input context
- Logit entropy at each generation step
These are weaker signals than activation variance
but measurable through standard LLM APIs.

Stability score computation:
- Raw signal: activation variance across L layers
- Normalize to 0.0 to 1.0 scale: 0.0 is maximally
  unstable, 1.0 is maximally stable
- Configurable stability threshold: default 0.7
- Below threshold: unstable, flag call site
- Above threshold: stable, proceed normally

Storage:
- Every LLM call record in PostgreSQL includes:
  stability_score float, stability_method enum
  (activation_variance, token_probability,
  perplexity, logit_entropy), stability_threshold
  float at time of measurement, stability_state
  enum (stable_pass, unstable_pass, stable_fail,
  unstable_fail)

Alembic migration:
- Add stability fields to all existing LLM call
  tables: llm_analysis, evaluations, any table
  that records LLM outputs

LiteLLM wrapper extension:
- Intercept all LLM calls
- Attempt activation variance measurement
- Fall back to token probability variance if
  activation unavailable
- Attach stability score to response before
  returning to caller
- Log stability event to audit.event Kafka topic

Dashboard:
- Stability monitor widget in Admin view
- Time series of stability scores per LLM call type
- Alert when sustained instability detected
- Stability state distribution chart: proportion
  of calls in each of four states over time
