# GeoLux Data Persistence

## Overview

GeoLux uses PostgreSQL as its primary data store, sharing the same database instance as Stargate. All GeoLux tables are prefixed with `glx_` to avoid collisions with existing Stargate tables. Schema management uses Alembic migrations (`alembic/versions/0001_initial_geolux_schema.py`). Kafka provides event streaming with topic-level retention policies.

ORM models are defined in `db/models.py`. CRUD operations are in `db/repository.py`. Database connection management is in `db/database.py` (pool_size=20, max_overflow=10, pool_pre_ping=True, pool_recycle=3600).

## 1. Table Reference

### 1.1 `glx_llm_stability_records` -- Geometric Stability Measurements

Per-LLM-call geometric stability measurements. Records every call made through the `StabilityAwareLLMClient` wrapper (`api/stability/wrapper.py`).

| Column | Type | Nullable | Description | Sensitivity |
|--------|------|----------|-------------|-------------|
| `id` | `Integer` | No (PK) | Auto-incrementing primary key | Internal |
| `call_id` | `String(255)` | No (unique, indexed) | UUID identifying the LLM call | Internal |
| `endpoint` | `String(100)` | No (indexed) | Logical endpoint name: `hypothesis_generation`, `deepfield_classification`, `mpc_prediction`, `mpc_optimization`, `semantic_classification` | Internal |
| `model` | `String(255)` | No | LLM model identifier (e.g., `granite-3-2-8b-instruct`) | Internal |
| `stability_score` | `Float` | No | Computed stability score, 0.0-1.0 | Operational |
| `stability_method` | `Enum(StabilityMethod)` | No | Method used: `activation_variance`, `token_probability`, `perplexity`, `logit_entropy` | Internal |
| `stability_threshold` | `Float` | No | Threshold applied at time of measurement | Operational |
| `stability_state` | `Enum(StabilityState)` | No | Four-state classification: `stable_pass`, `unstable_pass`, `stable_fail`, `unstable_fail` | Operational |
| `raw_signal` | `JSON` | Yes | Raw logprobs metadata (`{logprobs_length, latency_ms}`) | Operational |
| `created_at` | `DateTime(tz)` | No | Timestamp of measurement | Internal |

**Indexes**: `call_id` (unique), `endpoint`

**Access pattern**: Write-heavy. One record per LLM call. Read for stability trend analysis and threshold calibration via `GET /stability/scores`.

**Data sensitivity**: Low. Contains no PII. Stability scores could reveal LLM performance characteristics.

**Retention**: 90 days recommended.

### 1.2 `glx_hypotheses` -- Hypothesis Engine Records

Structured falsifiable hypotheses generated by THE (The Hypothesis Engine) from evidence bundles via `engine/hypothesis.py`.

| Column | Type | Nullable | Description | Sensitivity |
|--------|------|----------|-------------|-------------|
| `id` | `Integer` | No (PK) | Auto-incrementing primary key | Internal |
| `hypothesis_id` | `String(255)` | No (unique, indexed) | UUID identifying the hypothesis | Internal |
| `evidence_bundle_id` | `String(255)` | No (indexed) | Reference to the source evidence bundle | Internal |
| `claim` | `Text` | No | Natural language falsifiable claim about system state | Operational |
| `testable_conditions` | `JSON` | No | Array of conditions: `[{field, operator, value}]` where operator is one of `eq/gt/lt/gte/lte/contains/matches` | Operational |
| `confidence_score` | `Float` | No | LLM's confidence in the hypothesis (0.0-1.0); not used for decisions | Operational |
| `geometric_stability_score` | `Float` | No | Stability score of the LLM call that generated this hypothesis | Operational |
| `geometric_stability_method` | `Enum(StabilityMethod)` | No | Stability measurement method used | Internal |
| `geometric_stability_state` | `Enum(StabilityState)` | No | Stability state at generation time | Operational |
| `validation_outcome` | `Enum(ValidationOutcome)` | Yes | Deterministic result: `validated`, `falsified`, `inconclusive`; null when unvalidated | Operational |
| `validated_at` | `DateTime(tz)` | Yes | When validation occurred | Internal |
| `evidence_snapshot` | `JSON` | Yes | Snapshot of evidence fields at generation time | Operational |
| `stale` | `Boolean` | No | Whether this hypothesis was returned from cache during LLM unavailability (default false) | Internal |
| `created_at` | `DateTime(tz)` | No | Timestamp of generation | Internal |
| `audit_record_id` | `String(255)` | Yes (indexed) | Reference to the audit event that created this record | Internal |

**Indexes**: `hypothesis_id` (unique), `evidence_bundle_id`, `geometric_stability_score` (`idx_glx_hyp_stability`), `validation_outcome` (`idx_glx_hyp_validation`), `audit_record_id`

**Access pattern**: Write on evidence ingestion (1-N records per bundle). Read for hypothesis queue retrieval (unvalidated, ordered by stability score desc then confidence desc). Updated via `POST /hypotheses/{id}/validate` which sets `validation_outcome` and `validated_at`.

**Data sensitivity**: Low-Medium. `claim` and `evidence_snapshot` may reference infrastructure identifiers, cluster names, and namespace details. No direct PII unless evidence bundles contain operator-submitted text.

**Retention**: 90 days recommended.

### 1.3 `glx_constraint_definitions` -- Formal Constraint Definitions

Versioned constraint definitions synced from YAML files under `constraints/stages/`. Each defines a formal assertion against evidence for a specific provisioning stage.

| Column | Type | Nullable | Description | Sensitivity |
|--------|------|----------|-------------|-------------|
| `id` | `Integer` | No (PK) | Auto-incrementing primary key | Internal |
| `constraint_id` | `String(255)` | No (unique, indexed) | Stable identifier for the constraint | Internal |
| `constraint_name` | `String(255)` | No | Human-readable name | Internal |
| `stage` | `String(100)` | No (indexed) | Provisioning stage: `run-created`, `namespace-ready`, `deployment-ready`, `route-ready`, `vm-runtime-ready`, `storage-clone-ready`, `model-endpoint-ready`, `showroom-healthy`, `smoke-test-ready`, `provision-complete`, `cluster-health` | Internal |
| `assertion_type` | `Enum(AssertionType)` | No | Type: `threshold`, `boolean`, `range`, `pattern`, `composite`, `semantic` | Internal |
| `assertion_definition` | `JSON` | No | Definition of the assertion (field, operator, value for threshold/boolean; min/max for range; pattern string for pattern; nested assertions array + logic for composite; question + field for semantic) | Internal |
| `evidence_requirements` | `JSON` | No | List of evidence field names required for evaluation | Internal |
| `severity` | `Enum(Severity)` | No | Impact level: `critical`, `major`, `minor` | Internal |
| `remediation_class` | `String(255)` | Yes | Reference to a remediation procedure | Internal |
| `geometric_stability_weight` | `Float` | No | Weight for stability in confidence computation (default 0.5) | Internal |
| `version` | `Integer` | No | Schema version for compatibility checking (default 1) | Internal |
| `created_at` | `DateTime(tz)` | No | When definition was loaded | Internal |
| `deprecated_at` | `DateTime(tz)` | Yes | When definition was deprecated (null = active) | Internal |

**Indexes**: `constraint_id` (unique), `stage`

**Access pattern**: Bulk-loaded from YAML at startup or on first classification request via `constraints/loader.py:sync_constraints_to_db()`. Read on every classification. Active constraints filtered by `deprecated_at IS NULL`.

**Data sensitivity**: None. Configuration data only.

**Retention**: Indefinite (soft-delete via `deprecated_at`). Version history is valuable for audit and rollback.

**Retention**: 90 days recommended.

### 1.5 `glx_mpc_cycles` -- Model Predictive Control Cycles

Full records of MPC planning cycles including state, predictions, candidate actions, and optimization results, created by `engine/mpc.py:MPCController.plan()`.

| Column | Type | Nullable | Description | Sensitivity |
|--------|------|----------|-------------|-------------|
| `id` | `Integer` | No (PK) | Auto-incrementing primary key | Internal |
| `cycle_id` | `String(255)` | No (unique, indexed) | UUID identifying the MPC cycle | Internal |
| `cluster_id` | `String(255)` | No (indexed) | Target cluster for the planning cycle | Internal |
| `horizon` | `Integer` | No | Prediction horizon in steps (capped at `MPC_MAX_HORIZON`, default 5) | Operational |
| `current_state` | `JSON` | No | System state snapshot at cycle start | Operational |
| `predictions` | `JSON` | No | LLM-generated state predictions per step: `[{step, predicted_state, confidence}]` | Operational |
| `candidate_actions` | `JSON` | No | Generated candidate control actions: `[{action_id, action_type, parameters, score}]`. Action types: `scale_replicas`, `execute_remediation`, `no_action` | Operational |
| `selected_action_id` | `String(255)` | Yes | ID of the selected action (null if MPC suspended) | Operational |
| `optimization_score` | `Float` | Yes | Score of the selected action (0.0-1.0) | Operational |
| `geometric_stability_profile` | `JSON` | No | Stability profile: `{scores: [float], mean: float, min: float}` | Operational |
| `horizon_adjusted` | `Boolean` | No | Whether horizon was shortened/extended due to stability (default false) | Internal |
| `suspended` | `Boolean` | No | Whether MPC was suspended due to 3+ consecutive instabilities (default false) | Operational |
| `created_at` | `DateTime(tz)` | No | Timestamp of cycle | Internal |
| `audit_record_id` | `String(255)` | Yes (indexed) | Reference to the audit event | Internal |

**Indexes**: `cycle_id` (unique), `cluster_id`, `audit_record_id`

**Access pattern**: Write on each MPC planning cycle. Read for cycle history via `GET /mpc/cycles` and `GET /mpc/cycles/{cycle_id}` for full detail including predictions and candidate actions.

**Data sensitivity**: Low-Medium. `current_state` and `predictions` contain infrastructure state details that could reveal system architecture to unauthorized viewers.

**Retention**: 90 days recommended.

### 1.7 `glx_nano_obs_records` -- NanoObs Threshold Observations

Nano agent threshold observation and drift detection records, created by `engine/nanoobs.py:NanoObsCollector`. Tracks observed values against configured thresholds and records adaptive adjustment recommendations with human approval workflow.

| Column | Type | Nullable | Description | Sensitivity |
|--------|------|----------|-------------|-------------|
| `id` | `Integer` | No (PK) | Auto-incrementing primary key | Internal |
| `observation_id` | `String(255)` | No (unique, indexed) | UUID identifying the observation | Internal |
| `cluster_id` | `String(255)` | No (indexed) | Cluster being observed | Internal |
| `agent_id` | `String(255)` | No (indexed) | Nano agent identifier | Internal |
| `threshold_id` | `String(255)` | No | Threshold definition being observed | Internal |
| `observed_value` | `Float` | No | Actual measured value | Operational |
| `threshold_value` | `Float` | No | Configured threshold at observation time | Operational |
| `drift_detected` | `Boolean` | No | Whether statistical drift was detected (relative magnitude > 0.15 or std_dev > 30% of threshold; default false) | Operational |
| `drift_magnitude` | `Float` | Yes | Relative magnitude of drift: `(mean - threshold) / |threshold|` | Operational |
| `adjustment_recommended` | `Boolean` | No | Whether an adjustment was recommended (when drift detected and magnitude > 0.1; default false) | Operational |
| `adjustment_value` | `Float` | Yes | Recommended new threshold via exponential smoothing: `threshold + (observed - threshold) * 0.5` | Operational |
| `adjustment_approved` | `Boolean` | Yes | Whether a human approved the adjustment | Operational |
| `approved_by` | `String(255)` | Yes | **PII**: Identity of the person who approved the adjustment | PII |
| `approved_at` | `DateTime(tz)` | Yes | When the adjustment was approved | Internal |
| `created_at` | `DateTime(tz)` | No | Timestamp of observation | Internal |

**Indexes**: `observation_id` (unique), `cluster_id`, `agent_id`, composite `(cluster_id, agent_id)` (`idx_glx_nobs_cluster_agent`)

**Access pattern**: Write-heavy from continuous observation. Read for drift detection (last 10-20 records per agent via `detect_drift()`), adjustment recommendations (last 50 records via `recommend_adjustment()`), and drift summaries (last 100 records per cluster via `get_drift_summary()`). Updated on approval via `approve_adjustment()` which sets `adjustment_approved`, `approved_by`, `approved_at`.

**Data sensitivity**: Medium. `approved_by` is PII (operator identity who approved threshold adjustments).

**Retention**: 90 days recommended.

### 1.8 `glx_audit_events` -- Unified Audit Trail

Audit trail for all GeoLux component actions. Hash-chained for tamper detection via `api/security.py`.

| Column | Type | Nullable | Description | Sensitivity |
|--------|------|----------|-------------|-------------|
| `id` | `Integer` | No (PK) | Auto-incrementing primary key | Internal |
| `event_id` | `String(255)` | No (unique, indexed) | UUID identifying the audit event | Internal |
| `source_component` | `String(100)` | No (indexed) | Originating component: `hypothesis-engine`, `classification-engine`, `llm-mpc`, `deepfield`, `nanoobs` | Internal |
| `event_type` | `String(100)` | No (indexed) | Event type -- see complete list below | Internal |
| `payload_reference` | `String(255)` | Yes | Reference to the related entity (bundle_id, cluster_id, observation_id, composite path like `cluster/agent/threshold`) | Internal |
| `geometric_stability_score` | `Float` | Yes | Stability score at event time, if applicable | Operational |
| `operator` | `String(255)` | Yes | **PII**: Human operator identity associated with the event | PII |
| `trigger_type` | `Enum(TriggerType)` | No | How the event was triggered: `auto`, `manual`, `system` (default `auto`) | Internal |
| `created_at` | `DateTime(tz)` | No | Timestamp of the event | Internal |

**Event types**:
- `hypothesis.generation.started` -- hypothesis generation initiated
- `hypothesis.generation.gated` -- generation blocked by stability threshold breach
- `hypothesis.all_falsified` -- all hypotheses for a bundle falsified
- `stability.threshold.breach` -- stability score dropped below threshold
- `rubric.extension.triggered` -- novel failure mode suspected, human review needed
- `classification.started` -- classification run initiated
- `mpc.cycle.started` -- MPC planning cycle initiated
- `mpc.suspended` -- MPC suspended due to sustained instability
- `deepfield.routing.started` -- routing decision initiated
- `nanoobs.drift.detected` -- threshold drift detected
- `nanoobs.adjustment.recommended` -- adjustment recommendation generated
- `nanoobs.adjustment.approved` -- adjustment approved by human operator

**Indexes**: `event_id` (unique), `source_component` (`idx_glx_audit_source`), `event_type` (`idx_glx_audit_type`)

**Access pattern**: Write-heavy (every significant operation creates 1-3 audit events). Read for chain verification, incident investigation, and compliance reporting via `repository.get_audit_events()`.

**Data sensitivity**: High. `operator` field is PII -- contains the identity of humans who triggered manual actions (routing overrides, threshold adjustment approvals, manual scenario runs). This table is a primary target for GDPR data subject requests.

**Retention**: 1 year (365 days) recommended. Aligns with Kafka `geolux-audit-events` topic retention.

## 3. Data Lifecycle

### 3.1 Creation Patterns

| Table | Creation Trigger | Records Per Event |
|-------|------------------|-------------------|
| `glx_llm_stability_records` | Every LLM call through `StabilityAwareLLMClient.call()` | 1 per LLM call |
| `glx_hypotheses` | `POST /hypotheses/generate` via `engine/hypothesis.py:generate_hypotheses()` | 1-N per evidence bundle (one per LLM-generated hypothesis) |
| `glx_constraint_definitions` | First classification request or explicit sync via `constraints/loader.py:sync_constraints_to_db()` | One-time bulk load from YAML |
| `glx_classifications` | `POST /classify` via `engine/classification.py:classify_evidence()` | 1 per constraint per evidence bundle |
| `glx_mpc_cycles` | `POST /mpc/plan` via `engine/mpc.py:MPCController.plan()` | 1 per planning cycle |
| `glx_routing_decisions` | `POST /deepfield/route` via `engine/deepfield.py:DeepfieldRouter.route()` | 1 per routing request |
| `glx_nano_obs_records` | `engine/nanoobs.py:NanoObsCollector.observe()` | 1 per observation |
| `glx_audit_events` | Every significant operation across all components | 1-3 per operation (started + completed/gated, plus optional events like suspension or threshold breach) |
| `glx_launchpad_intelligence` | Launchpad intelligence collection | 1 per intelligence snapshot per time window |

### 3.2 Update Patterns

Most tables are append-only. The exceptions:

| Table | Updated Column(s) | Trigger |
|-------|-------------------|---------|
| `glx_hypotheses` | `validation_outcome`, `validated_at` | `POST /hypotheses/{id}/validate` via `repository.update_hypothesis_validation()` |
| `glx_nano_obs_records` | `adjustment_approved`, `approved_by`, `approved_at` | `NanoObsCollector.approve_adjustment()` |
| `glx_constraint_definitions` | `deprecated_at` | When a constraint definition is superseded by a newer version |

### 3.3 Delete Patterns

No tables have application-level delete operations. Data removal is handled exclusively via retention policies (see section 7).

### 3.4 Access Patterns Summary

| Query Pattern | Tables | Frequency | Method |
|---------------|--------|-----------|--------|
| Stability trend analysis | `glx_llm_stability_records` | Periodic dashboard | `repository.get_stability_records(endpoint, limit)` |
| Hypothesis queue retrieval | `glx_hypotheses` (unvalidated, ordered by stability then confidence desc) | Per evidence bundle | `repository.get_hypothesis_queue(evidence_bundle_id, limit)` |
| Hypothesis lookup | `glx_hypotheses` | Per validation | `repository.get_hypothesis(hypothesis_id)` |
| Bundle hypotheses | `glx_hypotheses` | Per bundle analysis | `repository.get_hypotheses_for_bundle(evidence_bundle_id, limit)` |
| Classification lookup | `glx_classifications` | Per classification | `repository.get_classification(classification_id)` |
| Active constraints | `glx_constraint_definitions` filtered by `deprecated_at IS NULL` | Per classification run | `repository.get_constraint_definitions(stage)` |
| MPC activation gating | `glx_classifications` filtered by cluster_id pattern and time window | Per MPC cycle | `MPCController.check_activation_gate()` |
| MPC cycle history | `glx_mpc_cycles` | Dashboard, post-mortem | `repository.get_mpc_cycles(cluster_id, limit)` |
| Routing history | `glx_routing_decisions` | Dashboard, audit | `repository.get_routing_decisions(workload_id, limit)` |
| Drift detection | `glx_nano_obs_records` by cluster_id + agent_id (last 10-20) | Continuous | `NanoObsCollector.detect_drift()` |
| Adjustment recommendation | `glx_nano_obs_records` by cluster_id + agent_id (last 50) | On drift detection | `NanoObsCollector.recommend_adjustment()` |
| Drift summary | `glx_nano_obs_records` by cluster_id (last 100) | Dashboard | `NanoObsCollector.get_drift_summary()` |
| Audit trail query | `glx_audit_events` by source_component or event_type | Investigation, compliance | `repository.get_audit_events(source_component, event_type, limit)` |
| Intelligence retrieval | `glx_launchpad_intelligence` by intelligence_type (last 10-50) | Dashboard | `repository.get_intelligence_records(intelligence_type, limit)` |

## 4. PII Identification

### 4.1 Fields Containing PII

| Table | Column | PII Type | How It Enters |
|-------|--------|----------|---------------|
| `glx_audit_events` | `operator` | Operator name/identity | From authenticated session: OIDC `sub`/`email`, proxy `X-Forwarded-User`, or parameter in NanoObs approval |

## 5. Encryption

### 5.1 Data at Rest

- **PostgreSQL TDE**: Transparent Data Encryption should be enabled on the PostgreSQL instance to encrypt all `glx_*` table data at rest. On OpenShift, this is typically achieved via LUKS encryption on persistent volumes.
- **Kubernetes Secrets**: Database credentials and API keys stored as Kubernetes Secret objects, encrypted by the cluster's etcd encryption configuration
- **Kafka**: Topic data at rest depends on the Kafka cluster's disk encryption configuration

### 5.2 Data in Transit

- **Application to PostgreSQL**: TLS via `sslmode=require` in the connection string (configure in `GEOLUX_DATABASE_URL`)
- **Application to Kafka**: TLS via broker configuration (`security.protocol=SSL` in producer config)
- **Application to LLM (LiteLLM)**: HTTPS via `GEOLUX_LITELLM_URL`
- **Application to Gaudi/Xeon6**: HTTPS via `GEOLUX_GAUDI_URL` / `GEOLUX_XEON6_URL`
- **Client to application**: TLS terminated at the OpenShift route; HSTS enforced via `Strict-Transport-Security: max-age=31536000; includeSubDomains` header

### 5.3 Field-Level Encryption

No field-level encryption is currently implemented. Sensitive fields (`operator`, `override_operator`, `approved_by`) are stored in plaintext within the encrypted-at-rest database. If regulatory requirements demand field-level encryption for PII, these three columns are the primary candidates.

## 6. Audit Event Hash Chain

### 6.1 How It Works

The audit trail implements a hash-chained integrity mechanism for tamper detection, implemented in `api/security.py`.

**Hash computation** (`compute_audit_hash()`):

```
hash = SHA-256( event_id + ":" + payload_reference + ":" + previous_hash )
```

- Each event's hash depends on the previous event's hash, forming a chain
- The first event in the chain uses an empty string as `previous_hash`
- Hash is computed as a hex digest of the SHA-256 output

**Chain verification** (`verify_audit_chain()`):

1. Walk events in chronological order
2. For each event, recompute the expected hash from `event_id`, `payload_reference`, and the previous event's hash
3. Compare against the stored `hash` field
4. If any mismatch is found, the chain is considered tampered

### 6.2 What Tampering Is Detected

- Modification of any event's `event_id` or `payload_reference`
- Deletion of events from the middle of the chain
- Insertion of forged events into the chain
- Reordering of events

### 6.3 Limitations

- Fields not included in the hash (`operator`, `geometric_stability_score`, `trigger_type`, `source_component`, `event_type`) can be modified without chain breakage
- Truncation of the chain from the end is not detectable (appending is indistinguishable from normal operation)
- If an attacker recomputes the entire chain from a tampered starting point, the recalculated hashes will appear valid

### 6.4 Verification Recommendations

- Run `verify_audit_chain()` periodically on the full event sequence
- Cross-reference database audit events with the Kafka `geolux-audit-events` topic as an independent append-only record
- Store periodic chain checkpoints (latest hash + event count) in an external tamper-evident store (e.g., a separate database, object storage with write-once policies, or a blockchain anchor)

## 8. Data Retention Policy Recommendations

### 8.1 Operational Tables (90-Day Retention)

These tables contain operational data that is primarily valuable for real-time and recent-history analysis:

| Table | Recommended Retention | Rationale |
|-------|----------------------|-----------|
| `glx_llm_stability_records` | 90 days | Stability trends are meaningful over weeks, not years |
| `glx_hypotheses` | 90 days | Validated/falsified hypotheses lose relevance as evidence evolves |
| `glx_classifications` | 90 days | Classification results are point-in-time; aggregate trends are derived; also used for MPC activation gating which requires `min_history_weeks` (default 4) of data |
| `glx_mpc_cycles` | 90 days | MPC cycles are meaningful for recent operational analysis |
| `glx_routing_decisions` | 90 days | Routing patterns are relevant for recent capacity planning |
| `glx_nano_obs_records` | 90 days | Drift detection uses last 20-50 records; historical records are archival |
| `glx_launchpad_intelligence` | 90 days | Intelligence snapshots are replaced by newer observations |

### 8.2 Audit Events (1-Year Retention)

| Table | Recommended Retention | Rationale |
|-------|----------------------|-----------|
| `glx_audit_events` | 365 days | Regulatory compliance, incident investigation, hash chain integrity. Aligns with the 365-day Kafka retention on `geolux-audit-events` |

### 8.3 Configuration Tables (Indefinite)

| Table | Recommended Retention | Rationale |
|-------|----------------------|-----------|
| `glx_constraint_definitions` | Indefinite (soft-delete via `deprecated_at`) | Version history is valuable for audit and rollback |

### 8.4 Replay Archives

Kafka replay archives (`geolux-replay-scenario` topic) retain for 7 days, consistent with the topic retention configuration.

### 8.5 Implementation

Retention should be implemented via a scheduled background job:

```sql
-- Run daily
DELETE FROM glx_llm_stability_records WHERE created_at < NOW() - INTERVAL '90 days';
DELETE FROM glx_hypotheses WHERE created_at < NOW() - INTERVAL '90 days';
DELETE FROM glx_classifications WHERE created_at < NOW() - INTERVAL '90 days';
DELETE FROM glx_mpc_cycles WHERE created_at < NOW() - INTERVAL '90 days';
DELETE FROM glx_routing_decisions WHERE created_at < NOW() - INTERVAL '90 days';
DELETE FROM glx_nano_obs_records WHERE created_at < NOW() - INTERVAL '90 days';
DELETE FROM glx_launchpad_intelligence WHERE created_at < NOW() - INTERVAL '90 days';
DELETE FROM glx_audit_events WHERE created_at < NOW() - INTERVAL '365 days';
```

For large-scale deployments, consider PostgreSQL table partitioning by `created_at` month with partition dropping instead of row-level deletes.

Kafka retention is enforced automatically by the broker based on `retention_ms` per topic.

## 9. GDPR and Data Subject Considerations

### 9.1 PII Scope

The following fields constitute personal data under GDPR when they contain identifiable operator information:

| Table | Column | GDPR Lawful Basis |
|-------|--------|-------------------|
| `glx_audit_events` | `operator` | Art. 6(1)(f) -- legitimate interest for operational audit |
| `glx_routing_decisions` | `override_operator` | Art. 6(1)(f) -- legitimate interest for routing accountability |
| `glx_routing_decisions` | `override_reason` | Art. 6(1)(f) -- may contain incidental PII |
| `glx_nano_obs_records` | `approved_by` | Art. 6(1)(f) -- legitimate interest for approval accountability |

### 9.2 Lawful Basis

Processing of operator names in audit trails is based on legitimate interest (Art. 6(1)(f)):
- Operational accountability for infrastructure decisions
- Security incident investigation
- Regulatory compliance for automated decision systems

### 9.3 Data Subject Rights

**Right to Access (Art. 15)**: Query all three PII-containing tables for a given operator name:

```sql
SELECT * FROM glx_audit_events WHERE operator = '<name>';
SELECT * FROM glx_routing_decisions WHERE override_operator = '<name>';
SELECT * FROM glx_nano_obs_records WHERE approved_by = '<name>';
```

**Right to Erasure (Art. 17)**: Erasure of operator names from audit events may conflict with the legitimate interest in maintaining audit integrity. Consider:
- Pseudonymizing operator names (replacing with an opaque identifier) rather than deleting entire records
- The audit event hash chain will not break from operator field modification since `operator` is not included in the hash computation (only `event_id` and `payload_reference` are hashed)
- Document the retention period (1 year for audit events, 90 days for operational tables) so erasure is time-bounded

**Right to Rectification (Art. 16)**: If an operator name was recorded incorrectly, update across all three tables. The audit hash chain is not affected since `operator` is not part of the hash.

### 9.4 Data Protection Impact Assessment

GeoLux's automated decision systems (Deepfield routing, MPC action selection) use LLM-based classification to make infrastructure decisions. Under Art. 35(3)(a), a DPIA is required when automated processing produces decisions that significantly affect individuals. Currently, GeoLux decisions affect infrastructure resources, not individuals directly, but the DPIA should be reviewed if:
- Decisions begin affecting individual user workloads
- Operator performance is evaluated based on override patterns
- Cost attribution data includes per-operator breakdowns

### 9.5 Kafka Considerations

Operator names flowing to Kafka topics are subject to the topic retention period:
- `geolux-audit-events`: 365-day retention (operator names in audit payloads)
- `geolux-deepfield-routing`: 30-day retention (override_operator in routing payloads)

Kafka does not support per-message deletion. GDPR erasure requests against Kafka data must either:
- Wait for retention-based expiry
- Use topic compaction with tombstone records keyed on the operator identity
- Rewrite affected topic partitions (operationally complex)

The bounded retention periods (30-365 days) provide a natural upper bound on PII persistence in Kafka.
